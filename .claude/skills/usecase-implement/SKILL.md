# Implement a Use Case

Adds behavior and/or automated validation for an ID from [usecases.md](../../../usecases.md). The endpoint of this workflow is: the ID passes `make usecase-validate USECASE=<ID>` and appears in the **Automated IDs** table of [docs/usecase-validation-matrix.md](../../../docs/usecase-validation-matrix.md).

## Scope check before starting

- **Planned IDs** (`C*`, `V*`, `T*`, `M*`, `D*`, `S*`, `P*`, `I*`) come from [masterplan.md](../../../masterplan.md) — implement freely within the architecture.
- **Adjacent IDs** (`AH*`, `AO*`, `AB*`, `AA*`) are not in the master plan. Confirm scope with the user before expanding the architecture; they may need a design note in [plans/](../../../plans) first.
- If the user is only asking to _promote_ an existing behavior into the validation gate (e.g. the feature already works, you're adding a test), skip to step 5.

## Core rules (non-negotiable)

From [CLAUDE.md](../../../CLAUDE.md) / [AGENTS.md](../../../AGENTS.md):

1. **No scenario-specific behavior in the Flutter client.** All new logic goes in the Go server. The client is a generic terminal.
2. **All client/server messages live in protobuf** under `api/terminals/<domain>/v1/*.proto` — never ad-hoc JSON. (Note: [CLAUDE.md](../../../CLAUDE.md) says `api/proto/` but the actual path is `api/terminals/<domain>/v1/` — trust the filesystem.)
3. **AI providers stay behind interfaces** in server code.
4. **UIs are assembled from shared server-driven primitives**, not scenario-specific widgets.

If your design requires violating any of these, stop and discuss with the user before writing code.

## Procedure

### 1. Ground the work
Re-read the target row in [usecases.md](../../../usecases.md) and the relevant plan file(s):
- [plans/use-case-flows.md](../../../plans/features/use-case-flows.md) — end-to-end flow for each planned scenario; start here.
- [plans/scenario-engine.md](../../../plans/features/scenario-engine.md) — scenario definitions, activations, triggers, preemption.
- [plans/application-runtime.md](../../../plans/features/application-runtime.md) — session lifecycle and orchestration.
- [plans/protocol.md](../../../plans/features/protocol.md) — gRPC + WebRTC + data channel shapes.
- [plans/server-driven-ui.md](../../../plans/features/server-driven-ui.md) — UI primitive set.

### 2. Write the failing test first (red)
Integration tests live in one of:
- `terminal_server/internal/transport/*_integration_test.go` — most scenarios.
- `terminal_server/cmd/server/*_test.go` — whole-binary behavior (photo frame config, silence classifier, due-timer loop, etc.).

**Name tests so both the Generated and Wire variants match a single regex** — this is how [scripts/usecase-validate.sh](../../../scripts/usecase-validate.sh) dispatches. Existing pairs you can model on:

| ID | Generated test | Wire test |
|---|---|---|
| C1 | `TestGeneratedSessionIntercomEmitsRouteStream` | `TestWireSessionIntercomEmitsRouteStream` |
| M3 | `TestGeneratedSessionRedAlertRelaysBroadcastNotification` | `TestWireSessionRedAlertRelaysBroadcastNotification` |
| S1 | `TestGeneratedSessionVoiceShowAllCamerasStartsMultiWindow` | `TestWireSessionVoiceShowAllCamerasStartsMultiWindow` |

The regex form in the script — `Test(Generated|Wire)Session<Name>$` — assumes you write both. A scenario that genuinely only has one form (e.g. a pure config/server-loop case like `D1`, `M1`, `T1`) can use a single test; match the pattern of the closest existing example.

Run the new test and confirm it fails for the right reason before implementing.

### 3. Update protobuf if needed
Any new client/server message must land in `api/terminals/<domain>/v1/*.proto` first. Regenerate and lint:
```bash
make proto-generate
make proto-lint
make proto-breaking   # catches incompatible changes against main
```
The Dart bindings under `api/gen/dart/` and Go bindings for the server are both regenerated by `buf`.

### 4. Implement server-side (green)
New scenarios and transport logic live under `terminal_server/`. Follow the existing package layout — `internal/transport/`, `cmd/server/`, etc. Run `make server-test` and `make server-lint` as you go. Iterate until the new test passes.

### 5. Wire the ID into validation
Three files must stay in sync. The script is the source of truth; CI runs it.

a. **[scripts/usecase-validate.sh](../../../scripts/usecase-validate.sh)** — add a case inside the `run_usecase()` switch, e.g.:
   ```bash
   C2)
     run_go_test ./internal/transport 'Test(Generated|Wire)Session<YourName>$'
     ;;
   ```
   If the ID should run under `USECASE=all`, also add it to the `ids=(…)` array at the bottom of the script.

b. **[docs/usecase-validation-matrix.md](../../../docs/usecase-validation-matrix.md)** — move the ID from the **Planned / Not Yet Automated** list into the **Automated IDs** table with its command and primary evidence. Keep the matrix aligned with the script.

c. **Test name(s)** must match the regex you just added.

### 6. Run the gates
```bash
make usecase-validate USECASE=<ID>
make all-check
```
`make all-check` runs `all-lint` (`server-lint` + `client-lint` + `proto-lint`), `all-test` (`server-test` + `client-test`), and `proto-breaking`. Both commands must pass before reporting done.

## Reporting

When finished, tell the user:
- The ID(s) implemented and/or newly automated.
- Protobuf files changed (if any).
- Server files and packages touched.
- New integration test name(s) and where they live.
- The three sync points (script switch, script `ids=()`, matrix table) — confirm all three were updated.
- Output of the two gate commands.

Do not mark the task done until `make usecase-validate USECASE=<ID>` and `make all-check` both pass.
