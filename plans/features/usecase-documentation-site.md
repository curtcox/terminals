# Use Case Documentation Site

Generate a browseable web site that documents every use case from
[`usecases/`](../../usecases/), driven by the same validation runs that gate
CI. The site is the single artifact a developer or user opens to see what the
system can do, what has been built, and where the defects are.

## Goals

1. One web page per use case, plus an index of all use cases.
2. Untested use cases are visually obvious (banner + index badge).
3. Use cases with detected bugs or failing assertions are visually obvious
   from the index — a cursory glance shows red.
4. Media (screenshots, recorded audio, recorded video, captured user-input
   traces) is produced by the same validation harness that proves the use
   case works. No hand-curated screenshots.
5. A developer can open a page to see exactly what is implemented, including
   the current rendered UI. A user can open the same page to learn what the
   system can do and how to do it.

## Non-Goals

- A separate tutorial / marketing site. The same pages serve both audiences.
- Hand-authored walkthroughs. Narrative comes from the scenario YAML and the
  validation harness — if the scenario doesn't capture it, the docs don't
  show it.

## Source of Truth

- Use case list and prose: [`usecases/*.md`](../../usecases/) (parsed; one
  H2 / ID per use case).
- Automated coverage: scenario YAMLs under
  [`terminal_server/internal/usecasevalidation/testdata/`](../../terminal_server/internal/usecasevalidation/testdata/)
  and the Go test files (`c1_test.go`, `aa4_test.go`, …).
- Join key: the stable use-case ID (`C1`, `M5`, `AA4`, …). A use case is
  "automated" iff a matching test/scenario exists and `make usecase-validate`
  exercises it; otherwise it is "untested".

## Capture Strategy — Server-Side Synthetic Frames

The clients are generic terminal renderers; the server already owns the
protobuf UI primitives. Capture happens server-side during validation:

- **Screenshots**: render the protobuf UI tree at each scenario step into a
  PNG using a headless renderer over the same primitives the Flutter client
  consumes. One frame per `expect` / `assert` step, named by step ID.
- **Video**: stitch the per-step frames into an MP4 (ffmpeg, fixed cadence)
  to show the use case end-to-end.
- **Audio**: capture the synthesized audio buffers the scenario emits (TTS
  output, sound effects, intercom audio) as WAV/OGG.
- **User-interaction trace**: emit a JSON timeline of input events the
  scenario injected (taps, voice utterances, sensor triggers) and render it
  alongside the video as a transcript panel.

This is faster than driving a real client, deterministic in CI, and exercises
the contract that the client is actually told to render. The trade-off is
explicit: these are not pixel-accurate Flutter screenshots. The doc page
labels them "rendered from server primitives" and links to the manual UI
audit (`terminal_server/internal/scenario/audit/`) where divergence is
checked.

## Build Pipeline

1. `make usecase-validate` runs the harness in `capture` mode (new flag),
   which writes media + a `result.json` per use case to
   `artifacts/usecases/<ID>/`.
2. A new generator (`scripts/build-usecase-site.py` or
   `terminal_server/cmd/usecase-site/`) reads:
   - `usecases/*.md` for prose / ID list,
   - `artifacts/usecases/<ID>/result.json` for pass/fail + media manifest,
   and emits static HTML+assets into `docs/usecases-site/`.
3. The Go server gains a `/docs/usecases` route that serves
   `docs/usecases-site/` (embedded via `embed.FS`) so the live deploy and
   the PR-reviewable artifact are the same bytes.
4. CI publishes `artifacts/usecases/` as a workflow artifact. The committed
   `docs/usecases-site/` is regenerated by `make usecases-site` and a
   drift check (`make usecases-site-check`) fails CI if it is stale —
   same pattern as `make usecases-index`.

## Page Layout

### Index (`/docs/usecases/`)

- Table grouped by family (matching `usecases/` files).
- Each row: ID, title, status badge, last-validated timestamp.
- Status badges, in priority order:
  - 🔴 **DEFECT** — most recent run failed or flagged a bug
  - ⚠️ **UNTESTED** — no automated scenario
  - 🟡 **STALE** — last successful run older than N days
  - 🟢 **PASSING**
- Index is sortable; defects and untested float to the top by default so
  a cursory glance surfaces problems.

### Per-use-case page (`/docs/usecases/<ID>.html`)

Header strip, full-width, color-coded:

- 🟢 "Validated on <date> — all assertions passed."
- 🔴 "Failed on <date>: <first failing assertion>. <N> defects open."
- ⚠️ "UNTESTED — no automated scenario exists for this use case."

Body sections:

1. **What it does** — prose from `usecases/<family>.md`.
2. **How to use it** — derived from the scenario's user-interaction trace:
   ordered list of user actions (tap X, say "…", wait for Y).
3. **What you see** — embedded MP4 (autoplay muted, loop) + scrubbable
   frame strip. Each frame links to its full PNG.
4. **What you hear** — embedded audio players for each captured clip.
5. **Defects** — if `result.json` lists failures, each is shown with the
   failing assertion, the frame at failure time, and a link to the
   bug-report entry if one exists.
6. **Evidence** — links to the scenario YAML, the Go test, and the raw
   `result.json`.

### Untested pages

Generated for every ID with no scenario, so the index link works. Body is
the prose + a prominent "No automated validation. Behavior shown here is
specified, not verified." banner. No media section.

## Defect Surfacing

A use case is "in defect" if any of the following hold for its latest run:

- A Go test in `usecasevalidation/` referencing the ID failed.
- The scenario YAML asserted an `expect` that did not match.
- A bug report under [`docs/bug-reporting.md`](../../docs/bug-reporting.md)
  flow has an open entry tagged with the ID.

The generator merges these into `result.json` so the index and header use
one source. The bug-reporting tag is the integration point with the
existing `bugword` workflow.

## Milestones

1. **M1 — Skeleton.** Generator reads `usecases/` only, emits index + stub
   pages, no media. Server route serves it. Drift check wired into CI.
2. **M2 — Pass/fail surfacing.** Validation harness writes `result.json`
   per ID. Index shows green/red/⚠️. No media yet.
3. **M3 — Screenshots.** Harness captures per-step PNGs from server-side
   render. Per-page frame strip + lightbox.
4. **M4 — Video + audio.** ffmpeg stitch + audio capture. Embedded
   players.
5. **M5 — Interaction trace.** JSON timeline → "How to use it" section.
6. **M6 — Defect feed.** Join bug-report entries by ID; show on page +
   index.

Each milestone is independently shippable and leaves the site usable.

## Open Questions

- **Audio capture format and licensing.** TTS output from external
  providers may not be redistributable. Decide per-provider before M4.
- **Frame budget.** A long scenario could produce hundreds of PNGs.
  Default to one frame per `expect` step; allow scenarios to opt into
  finer cadence.
- **Cross-client divergence.** Server-side renders won't catch Flutter /
  macOS visual bugs. The `ui-inspect` skill already covers this — link
  out from each page rather than duplicating its captures here.
