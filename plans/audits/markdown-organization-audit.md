# Markdown Organization and Automation Audit

Date produced: 2026-05-01
Verifier: Claude Code session (read-only survey)
Scope: every `*.md` under the repo excluding `.git/`, `build/`, `.dart_tool/`,
`.pub-cache/`, `.sdk/`, `node_modules/`. 172 files in scope at the time of
audit.

## Why this audit exists

The repo already has strong machinery for plans (YAML frontmatter, status
vocabulary, `plans/INDEX.md` auto-generated by
[`scripts/generate-plans-index.py`](../../scripts/generate-plans-index.py),
and a `pick-next-work.py` ranker). That pattern is valuable but only applied
to one slice of the docs. Other slices — use cases, docs guides, top-level
nav, progress logs, the "what to do next" pointer — are still hand-maintained
and drift against each other.

This audit catalogs the highest-payoff reorganizations and lists scripts and
skills that would close the loop on **recording completed work** and
**identifying what to do next**, both of which the user explicitly called out
as goals.

The findings are ordered so they can be enacted incrementally: each finding
lists prerequisites and is independently shippable.

## Findings

### F1 — `docs/usecase-validation-matrix.md` is hand-maintained but every fact already exists in machine-readable form

**Severity:** high (drift compounds every time a use case is added or a plan
ships).

**Where:** [`docs/usecase-validation-matrix.md`](../../docs/usecase-validation-matrix.md)
(58 lines, ~26 entries today).

**Evidence:**

- Use-case IDs are defined in [`usecases.md`](../../usecases.md) (~70 IDs).
- Plans declare validation against IDs in YAML frontmatter:
  `validation: automated:<ID>` (already validated by
  [`scripts/generate-plans-index.py`](../../scripts/generate-plans-index.py)
  and surfaced in [`plans/INDEX.md`](../INDEX.md)).
- The matrix re-tabulates the join by hand, plus a "Planned / Not Yet
  Automated" prose section that lists IDs not present elsewhere.

**Risk:** silent divergence — a plan can flip to `automated:M5` without the
matrix being updated, or a use case can be deleted without the matrix
catching it. There is no test today that joins the two sources.

**Action:**

1. Replace `docs/usecase-validation-matrix.md` with a generated artifact:
   `scripts/generate-validation-matrix.py` (parses `usecases.md` for IDs and
   walks `plans/` frontmatter for `validation: automated:<ID>`). Wire as
   `make validation-matrix`.
2. While generating, emit two drift signals to stderr (and fail the build):
   - **Orphan use cases**: ID in `usecases.md`, no plan references it via
     `automated:<ID>`. (Today this list is hand-maintained at the bottom of
     the matrix file — auto-generate it.)
   - **Broken validation refs**: plan declares `automated:X`, but `X` is not
     defined in `usecases.md`.
3. Add `make validation-matrix` to the existing `make all-check` chain, or at
   least to a CI lint job, so drift is caught at PR time.

**Prereqs:** none — can be done first, before the `usecases.md` split (F2).

### F2 — `usecases.md` is a 222-line monolith that should be one file per family

**Severity:** medium (slows agent and human lookup; blocks frontmatter-based
automation).

**Where:** [`usecases.md`](../../usecases.md), 12 sections:
Communication (C), Voice (V), Timers (T), Monitoring (M), Display (D),
Security (S), Productivity (P), Infrastructure (I), Bug Reporting (B),
Adjacent-Home (AH), Adjacent-Office (AO), Adjacent-Business (AB), Software
Agents (AA), PLATO-inspired (PL).

**Evidence:**

- Every family section is a self-contained markdown table.
- Cross-family references in the file are essentially zero (each family is
  independent prose).
- Today, an agent answering "what does C2 cover?" must load the entire 222
  lines.

**Action:**

1. Split into `usecases/` directory: one file per family
   (`usecases/communication.md`, `usecases/voice.md`, …) with frontmatter:

   ```yaml
   ---
   title: "Communication Use Cases"
   family: C
   ids: [C1, C2, C3, C4, C5, C6]
   ---
   ```

2. Keep the family table verbatim — this is a pure split, not a rewrite.
3. Add `scripts/generate-usecases-index.py` (mirroring
   `generate-plans-index.py`) → `usecases/INDEX.md` listing each family and
   its IDs. Wire as `make usecases-index`.
4. Replace `usecases.md` at the repo root with a thin pointer to
   `usecases/INDEX.md`, or keep it as the index itself (rename target file).
5. Update `docs/usecase-validation-matrix.md` references (or, if F1 is done
   first, the new generator) to walk `usecases/*.md` instead of grepping a
   single file.

**Prereqs:** ideally do F1 first so the matrix generator already exists when
the data layout changes — easier to update one consumer than to migrate
data and consumer simultaneously.

### F3 — Progress logs are devouring the plans they live in

**Severity:** high (single biggest readability problem; affects the
plans most actively worked on).

**Where:**

- [`plans/features/app-migrations.md`](../features/app-migrations/plan.md) — 98KB,
  100 dated bullets in `## Implementation Progress`. Status `building`.
- [`plans/features/repl-and-shell.md`](../features/repl-and-shell.md) — 33KB,
  6 dated entries.
- [`plans/features/io-abstraction.md`](../features/io-abstraction/plan.md),
  [`plans/features/package-format.md`](../features/package-format/plan.md),
  [`plans/features/protocol.md`](../features/protocol.md),
  [`plans/features/world-model-calibration.md`](../features/world-model-calibration/plan.md)
  — all carry inline progress logs.

**Evidence:**

- The plan body (durable design, gates, scope) and the progress log
  (append-only, dated) have different lifecycles but live in the same file.
- The `repl-capability/` directory already demonstrates the better pattern:
  `plan.md` (durable) + `closure.md` (drained items) +
  `deferred.md` (deferred items). See
  [`plans/features/repl-capability/`](../features/repl-capability/).

**Action:**

1. For each plan with a progress log, convert to a folder:
   `plans/features/<plan>/plan.md` (durable design) +
   `plans/features/<plan>/progress.md` (append-only dated bullets).
2. Update `pick-next-work.py` and `generate-plans-index.py` to walk one level
   deeper — both already use `os.walk` so this should already work; verify
   they pick up `plan.md` correctly and skip `progress.md`
   (`progress.md` should not have plan-kind frontmatter).
3. Migrate `app-migrations.md` first — it has the highest mass and is
   actively `building`, so the win is immediate.

**Prereqs:** none. F3 can be done independently and incrementally
(one plan at a time).

### F5 — `next.md` is a hand-edited dangling pointer

**Severity:** low.

**Where:** [`next.md`](../../next.md), one sentence today, points at one
sub-task inside one plan.

**Evidence:**

- [`scripts/pick-next-work.py`](../../scripts/pick-next-work.py) already
  computes "what to work on next" by priority bucket.
- `next.md` is convention from `CLAUDE.md` but is updated by humans, not
  by the script.

**Action:** delete `next.md` and rely on `make pick-next-work`, or have
`pick-next-work.py` write its top recommendation to `next.md` so the file
stays accurate without human upkeep. Recommend the second — preserves the
existing convention without the staleness.

**Prereqs:** none.

## Recommended new scripts and skills

These deliver on the user's stated goals: **record completed work** and
**identify what to do next**.

### S1 — `/log-progress` skill

**Goal:** standardize how completed work is recorded so progress logs stay
machine-readable and `last-reviewed` stays accurate.

**Behavior (proposed):**

- Args: plan path (or plan title) + one-paragraph summary of what landed.
- Effects:
  1. Append a dated bullet to that plan's `progress.md` (or
     `## Implementation Progress` while F3 is in flight).
  2. Bump `last-reviewed:` in the plan's frontmatter to today's date.
  3. Optionally transition status (e.g. `building` → `shipped-untested`)
     if the user passes a flag.
  4. Re-run `make plans-index` so the index reflects the new state.

**Why it's a skill, not just a script:** the operation is conversational
(LLM picks the right plan, summarizes from the diff) but the file mutations
must be deterministic and gated.

---

### S2 — `/next` skill (or `make next`)

**Goal:** one canonical answer to "what should I work on?" replacing
`next.md`.

**Behavior (proposed):** wraps `pick-next-work.py` and additionally surfaces:

- Un-validated planned use-case IDs (those in `usecases.md` with no
  `automated:<ID>` reference in any plan).
- Open audits and incidents.
- Plans whose progress log has not been updated in N days but whose status
  is `building` (stalled signal).

**Prereqs:** F1 (so the use-case→validation join is reliable) makes the
"un-validated planned IDs" bucket sound.

---

### S3 — `make orphans` script

**Goal:** surface drift quickly. Lists:

- Use-case IDs not referenced by any plan via `automated:<ID>`.
- Plans declaring `automated:<ID>` where the ID does not exist.
- Docs not linked from `docs/README.md` (after F6).
- Plans in `plans/INDEX.md` whose frontmatter is malformed (already covered
  by `generate-plans-index.py`'s error mode — extend rather than duplicate).

**Prereqs:** F1 (validation matrix generator is the natural home for the
first two checks).

---

### S4 — `validate-frontmatter` lint, extended to docs and use cases

**Goal:** keep all of the above honest.

**Action:** the existing `generate-plans-index.py` already exits non-zero on
malformed frontmatter. Extend the same contract to `usecases/*.md` (after
F2) and `docs/*.md` (after F6). Add a single make target,
`make frontmatter-lint`, and include it in `make all-check`.

---

## Recommended ordering

1. **F1** (validation matrix generator) — blocks nothing, removes one of the
   biggest hand-maintained files, and gives F2 a consumer to test against.
2. **F3** (progress-log split) on `app-migrations.md` first — biggest
   single-file readability win.
3. **F2** (usecases split) — once F1 exists, the matrix generator just
   updates its source walk.
4. **S1** (`/log-progress` skill) — depends on F3 to land for the
   highest-value plans.
5. **S2** (`/next` skill) — depends on F1 for "un-validated IDs" bucket.
6. **F4 / F5 / F6 / F7 / F8** — independent low-severity cleanups, can be
   done in any order.
7. **S3 / S4** — fold in once the generators are stable.

## Out of scope

- Rewriting plan content. This audit is structural; the prose stays.
- The `docs/repl/` subtree. It already follows a sensible per-command
  decomposition.
- The `plans/archive/` and `plans/incidents/` subtrees. Both are small and
  follow the existing conventions.

## Closure criteria

This audit is `resolved` when:

- `docs/usecase-validation-matrix.md` is auto-generated and CI fails on
  drift (F1).
- At least the active `building` plans (today: app-migrations) have moved
  to the `<plan>/plan.md` + `<plan>/progress.md` shape (F3).
- A `/log-progress` skill exists and is documented in `SKILLS.md` (S1).
- A `/next` skill (or `make next`) exists and `next.md` is either
  auto-written or removed (S2 + F5).

The remaining findings (F2, F4, F6, F7, F8, S3, S4) can be resolved as
follow-ups without blocking closure.
